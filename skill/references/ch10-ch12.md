# Chapter 10 — Batch Processing

Batch processing: process a large amount of accumulated data on a schedule (vs stream = real-time).

---

## Batch Processing with Unix Tools

Unix philosophy: small programs that do one thing, composable via pipes. Input/output are streams of text (often newline-delimited).

Example: `cat access.log | awk '{print $7}' | sort | uniq -c | sort -rn | head -5`

Key properties:
- **Immutable input**: Commands read input without modifying it
- **Output to another program**: Programs are loosely coupled via stdin/stdout
- **Uniform interface**: All programs use text streams

These properties make Unix tools easy to compose, test, and reason about. **MapReduce applies the same principles at distributed scale.**

---

## MapReduce and Distributed Filesystems

**HDFS** (Hadoop Distributed File System): Distributed version of a Unix filesystem. Files replicated across multiple machines (default 3×). Very large files (petabyte scale).

**MapReduce**: Programming model for processing large datasets across a cluster.

1. **Map**: For each input record, emit zero or more key-value pairs.
2. **Sort/Shuffle**: Framework sorts all key-value pairs by key, groups by key, sends each group to a reducer.
3. **Reduce**: For each key, process all its values and emit output records.

**Characteristics**:
- Map and reduce functions are pure (no side effects). Framework can retry on failure.
- Input data is read from HDFS, output written to HDFS. The intermediate shuffle data is local.
- Each map/reduce task processes data independently.
- The framework handles parallelism, fault tolerance, and data movement.

**Distributed joins**:
- *Sort-merge join*: Both datasets are partitioned and sorted by join key. Merge matching keys in the reducer.
- *Broadcast hash join*: Small dataset fits in memory → broadcast to all mappers → each mapper probes hash table.
- *Partitioned (bucketed) hash join*: Both datasets partitioned the same way → each mapper only joins its partition.

**Workflows**: Chained MapReduce jobs (output of one is input of the next). Managed by workflow tools: Oozie, Azkaban, Luigi, Airflow.

**Handling skew**: Some keys have vastly more data (hot keys). Techniques: secondary sort, partial reducers, random sampling to identify hot keys.

---

## Beyond MapReduce

MapReduce problems:
- Materializes intermediate state to HDFS after every job — expensive
- Many workflows require many chained jobs (hundreds for ML pipelines)
- Slow due to replication and disk I/O of intermediate state

**Dataflow engines**: Spark, Flink, Tez — execute workflows as a DAG of operators. Operators communicate via in-memory buffers (not HDFS). Much faster.

**Operator types**: map, filter, join, group-by, sort, union, etc.

**Fault tolerance**: Unlike MapReduce (intermediate state on disk), dataflow engines must re-compute from the last materialized checkpoint. Lineage tracking (Spark RDDs) enables this.

**High-level APIs**: Hive, Pig (MapReduce), Spark SQL, Flink SQL — declarative APIs that compile to dataflow operators. Enable query optimization.

**Graph processing**: Pregel model (Google) / Spark GraphX — specialized bulk-synchronous parallel (BSP) computation for graph algorithms (PageRank, shortest path, connected components).

**Machine learning on batch**: MLlib (Spark), TensorFlow — batch training over large datasets.

---

## Principles of Batch Processing

- Input is immutable, output is deterministic given the same input
- Failures can be retried safely
- Separation of logic from the framework (pure functions)
- Output can be used as input to other jobs or loaded into serving stores

# Chapter 11 — Stream Processing

Batch processes: finite input, produce output and stop. Stream processes: input is unbounded (arrives continuously), process it as it arrives.

---

## Transmitting Event Streams

**Event**: A small self-contained, immutable record of something that happened at a point in time. Usually serialized as JSON, Avro, Thrift, etc.

**Messaging systems**: Connect event producers to consumers.

**Direct messaging** (WebSockets, UDP multicast, ZeroMQ): Producer sends directly to consumer. Low overhead. Consumer must be up; no buffering.

**Message brokers** (queues): Producers write to broker; consumers read from broker. Broker provides durability and buffering. Consumers can be offline. Examples: RabbitMQ, ActiveMQ.

**Log-based message brokers** (Kafka): Events are appended to a log on disk. Consumers track their offset. Retaining the log allows consumers to replay. Kafka partitions the log across multiple machines for scalability.

**Comparison**:
| | Traditional broker | Log-based broker (Kafka) |
|---|---|---|
| Delivery | Delete on ack | Append-only; consumer tracks offset |
| Ordering | Not guaranteed across consumers | Ordered within partition |
| Replay | Not possible after ack | Yes, by resetting offset |
| Consumer groups | Competing consumers | Each group gets its own offset |
| Use cases | Task queues, RPC | Event sourcing, CDC, analytics |

---

## Databases and Streams

Database writes are events. CDC (Change Data Capture) exposes them as a stream.

**CDC (Change Data Capture)**: Capture all writes to a database as a stream of change events. Use the stream to populate other systems (search indexes, caches, data warehouses) in real time.

CDC is typically derived from the replication log (logical decoding in PostgreSQL, binlog in MySQL).

**Event sourcing**: Don't store current state; store the log of all events. Derive current state by replaying events. Powerful for audit trails, time-travel, debugging. Requires careful schema evolution (events are immutable and forever).

**State snapshots**: Periodically snapshot derived state so you don't need to replay from the beginning of time.

**CQRS** (Command Query Responsibility Segregation): Separate read and write models. Write model is the event log; read models are derived views optimized for queries.

---

## Processing Streams

**Stream processing operators**: filter, map, group-by, join, aggregate, window.

**Time semantics**:
- *Processing time*: Time when the event is processed. Simple but misleading (late events skew results).
- *Event time*: Time when the event actually occurred. More accurate. Requires handling late events.

**Windows**:
- *Tumbling window*: Fixed-duration, non-overlapping (e.g., 1-hour buckets).
- *Hopping window*: Fixed-duration, overlapping (e.g., 5-min window every 1 min).
- *Sliding window*: All events within a time interval of each other.
- *Session window*: Groups events from the same user within an inactivity gap.

**Watermarks**: Estimate of "how far behind" processing is. Events with timestamp before the watermark are considered "complete." Used to trigger window computations.

**Stream-stream joins**: Match events from two streams within a time window (e.g., search query + click).

**Stream-table join (stream enrichment)**: Enrich events by looking up records in a database (join with a slowly changing table). Table must be replicated locally or loaded into memory.

**Table-table join**: Maintain a materialized view that continuously reflects joins between two CDC streams.

**Fault tolerance**: Micro-batching (Spark Streaming), checkpointing state to durable storage (Flink), transactional producers/consumers (Kafka Streams). Exactly-once semantics achievable within a single system; harder across systems.

**Stream processing frameworks**: Apache Kafka Streams, Apache Flink, Apache Spark Streaming, Apache Storm, Samza, Google Dataflow / Apache Beam.

## Key Concepts

| Term | Definition |
|---|---|
| Event | Immutable record of something that happened |
| Message broker | System for buffering and delivering events |
| Log-based broker | Events stored in append-only log; consumers track offsets (Kafka) |
| CDC | Change Data Capture — stream of DB writes |
| Event sourcing | Store the log of events as the system of record; derive state by replay |
| CQRS | Separate read and write models |
| Tumbling window | Fixed, non-overlapping time windows |
| Watermark | Estimate of event-time completeness for window triggering |
| Stream-table join | Enrich stream events with current database state |
| Exactly-once processing | Every event affects output exactly once despite failures and retries |

# Chapter 12 — The Future of Data Systems

---

## Data Integration

No single tool is best for all uses. Systems are composed: Kafka for events, Elasticsearch for search, Redis for cache, Snowflake for analytics, PostgreSQL for transactions.

**Derived data systems**: Some systems hold the system of record (source of truth); others hold derived data (indexes, caches, views). Derived data must be kept in sync.

The solution: treat the event log as the integration medium. Kafka acts as the "central nervous system" — all systems subscribe to the event stream and maintain their own derived views.

---

## Unbundling Databases

Traditional monolithic databases bundle together: storage, indexing, querying, replication, transactions, backup, etc.

The "unbundled database" idea: compose these features from specialized tools:
- Storage: HDFS, S3, cloud object storage
- Indexing: Elasticsearch, Lucene
- Stream processing: Kafka, Flink
- OLAP: Redshift, BigQuery
- Transactions: PostgreSQL, Spanner

The application connects them via CDC and event streams.

---

## Aiming for Correctness

Even with ACID databases, bugs and edge cases cause incorrect results. Beyond transactions: end-to-end correctness.

**Idempotent operations**: Safe to retry without side effects. Database operations that check a condition before applying (e.g., `UPDATE SET ... WHERE expected_value = x`).

**Exactly-once semantics**: Deduplication using unique request IDs at the application level. Even if the network times out and the request is retried, the operation must only be applied once.

**Constraints and integrity**: Cross-partition uniqueness constraints are hard in distributed systems. Techniques: async constraint checking with compensating actions, or funneling all writes through a single coordinator.

**Audit logs / event sourcing**: Complete history makes it possible to verify correctness and detect bugs retroactively.

---

## Doing the Right Thing

Data systems have ethical implications:

- **Privacy**: Collecting data creates risk; data can be stolen, subpoenaed, or misused. Minimize data collection. Allow deletion (GDPR right to erasure). Event logs that contain personal data are hard to purge.
- **Surveillance**: Behavioral tracking enables predicting and manipulating behavior.
- **Bias and discrimination**: ML systems trained on historical data can perpetuate biases. Feedback loops amplify initial biases.
- **Accountability**: Automated decisions that affect people (credit, insurance, hiring) should be explainable and contestable.
- **Predictive policing**: Historical crime data reflects policing patterns, not crime rates.

Engineers who build data systems bear responsibility for how they are used. "Move fast and break things" is not acceptable when it breaks people's lives.

## Key Themes Across the Book

1. **Reliability** requires thinking about hardware, software, and human faults.
2. **Scalability** requires understanding your specific load parameters and growth pattern.
3. **Maintainability** requires abstractions that hide complexity.
4. **Correctness** in distributed systems is hard but achievable with the right primitives.
5. **Data integration** across diverse tools requires treating the event log as truth.
6. **Ethics** is not optional — data engineers must consider the societal impact of their systems.
